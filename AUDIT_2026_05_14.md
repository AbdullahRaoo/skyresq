## Deferred â€” important, not blocking bench tests

### Drop altitude not handled

`sar_orchestrator._tick_approach` publishes fly_to with `altitude=NaN`
(meaning "hold current"). For a real outdoor mission the drone is at
the search altitude (~30 m) â€” too high for a useful drop. Need a
`drop_altitude_m` param (default 5 m AGL) and a DESCEND state between
APPROACH and DROP.

**Skip for now because:** props are off, no real flight planned this
week. Will surface again once SITL is set up.

### RTL state has no liveness timeout

`_tick_rtl` waits for `drone_lat` within `home_tol_m` of `home_lat`. If
`/vehicle/home` never publishes (bench, no GPS lock), RTL stays in
state forever. Should add `rtl_max_s` (e.g., 120 s) â†’ DONE.

### Orchestrator aborts on transient mode hiccups

In `APPROACH/DROP/DROP_HOLD`, any tick where `_mode != "GUIDED"` drops
to IDLE. If the FC briefly reports a different mode (GUIDED_NOGPS,
internal mode change race), we lose the mission. Need a grace period
(~2 s of confirmed wrong-mode before abort).

### Detector confidence 0.30 generates too many false positives

Earlier bench saw "Detected 10 person(s) | best conf: 0.81" â€” many low-
confidence detections cluttering the frame. 0.30 was set to catch a
seated user in low light; for the field demo a higher value (0.45â€“0.50)
trades sensitivity for stability. Tune at demo site.

## Deferred â€” nice-to-have

- **DHCP reservation on Blaze router**: pin Pi's MAC â†’ 192.168.1.139 so the
  dashboard's "Pi LAN host" never needs editing.
- **NTP on Pi**: `timedatectl set-ntp true` so detection_frame timestamps
  stay aligned with dev PC (1.5 s `DETECTION_STALE_MS` window assumes synced
  clocks).
- **journald rotation policy**: services log to journald, which by default
  is bounded but could grow. `sudo journalctl --vacuum-size=200M` once
  in a while; or set `SystemMaxUse=200M` in `/etc/systemd/journald.conf`.
- **Per-node CPU pinning**: ncnn detector pegs ~80% of one core; pinning
  via systemd `CPUAffinity=` could prevent it from competing with the
  more time-sensitive gimbal_controller.
- **gcs_link health field for orchestrator**: when `gcs_link` is up but
  GCS hasn't heard from us in N seconds, ArduPilot's `FS_GCS_ENABLE` will
  RTL anyway. Pi-side we should never assume the link is alive â€” fine
  to leave to FC failsafe.
